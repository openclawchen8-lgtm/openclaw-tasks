## 診斷結果

### 環境資訊
| 項目 | 值 |
|------|-----|
| CLI 版本 | clawhub v0.9.0（npm） |
| Config 路徑 | ~/Library/Application Support/clawhub/config.json |
| Token 前綴 | clh_（ClawHub 格式） |
| Token 產生日期 | 2026-04-03 12:10 |
| OS | macOS (Apple Silicon) |

### 問題分類：**雙重問題 — Issue #1717（P0）+ Token 失效**

本地問題同時命中兩個根因：

#### 問題 1：服務端 OAuth 端點崩潰（對應 #1717）
```bash
$ curl -s -o /dev/null -w "HTTP %{http_code}" https://clawhub.ai/api/auth/signin/github
HTTP 500
```
- `/api/auth/signin/github` 回傳 HTTP 500，空的 response body
- **完全無法透過瀏覽器重新登入取得新 token**
- 這是 ClawHub 服務端問題（AUTH_GITHUB_ID/SECRET 缺失或過期）
- **本地無法修復**，需 ClawHub 官方在 Convex dashboard 修復

#### 問題 2：現有 Token 已失效
```bash
$ curl -s -H "Authorization: Bearer <token>" https://clawhub.ai/api/v1/whoami
Unauthorized  # HTTP 401
```
- Token 產生於 2026-04-03，目前回 401 Unauthorized
- 可能原因：Token 過期、被撤銷、或對應帳號被 soft-delete
- 由於問題 1 存在，**無法透過 `clawhub login` 取得新 token**

### 影響範圍
- ❌ `clawhub whoami` → Unauthorized
- ❌ `clawhub publish` → Unauthorized
- ❌ `clawhub login` → HTTP 500（服務端崩潰，無法取得新 token）
- ✅ `clawhub search` → 正常（公開端點）
- ✅ `clawhub explore` → 正常（公開端點）
- ✅ `clawhub install` → 正常（公開端點）

## 修復方案

### 可立即執行（本地端）

#### Step 1：清除失效 Token
```bash
rm ~/Library/Application\ Support/clawhub/config.json
```
或使用 CLI：
```bash
clawhub logout
```

#### Step 2：等待服務端修復後重新登入
```bash
# 先驗證服務端已恢復
curl -s -o /dev/null -w "HTTP %{http_code}" https://clawhub.ai/api/auth/signin/github
# 期望: HTTP 200

# 服務端恢復後重新登入
clawhub login
```

### 需要 ClawHub 官方處理（本地端無法解決）

1. **修復 AUTH_GITHUB_ID / AUTH_GITHUB_SECRET**：在 Convex dashboard 重新設定 GitHub OAuth credentials（PR #1720 的 fail-fast 機制有助於未來更早發現此問題）
2. **確認 Token 失效原因**：是過期？被撤銷？還是帳號 soft-delete？如果與 #1116/#886 同類問題相關，需管理員清除舊記錄

### 臨時替代方案（如急需發布 skill）

如果急需使用 publish 功能，可嘗試：
1. 手動從 GitHub Settings 撤銷 ClawHub OAuth 授權
2. 服務端恢復後立即 `clawhub login`
3. 或等待官方提供新的 token 取得方式（如 PAT）

## 結論

**問題定位**：本地屬於 **P0 級別（Issue #1717）+ Token 失效**的複合問題。

**核心矛盾**：Token 已失效，但取得新 Token 的途徑（browser login）因服務端崩潰而完全不可用。這是一個「死循環」——需要登入才能修復登入問題。

**建議行動**：
1. 立即執行 `clawhub logout` 清除失效 token
2. 持續監控 `/api/auth/signin/github` 是否恢復 200
3. 恢復後立即 `clawhub login` 重新取得 token
4. 可考慮在 #1717 或 #1717 下的 comment 中追蹤修復進度
